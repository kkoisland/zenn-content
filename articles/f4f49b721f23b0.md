2023年秋頃、個人開発でAIを使い始めました。最初は「AIがあれば何でもできる」と思っていましたが、現実は違いました。試行錯誤を繰り返す中で気づいたのは、設計の重要性です。2025年夏頃から今のスタイルに落ち着いたので、本記事ではそのスタイルを紹介します。

## 使っているAI

現在、2つのAIを併用しています。

### ChatGPT（Plus / GPT-5.2 $20）

**向いていること**
- 何でも気軽に相談できる
- レスポンスが早く、思考の壁打ちに向いている
- 技術以外の雑談・整理・文章化にも使いやすい

**苦手なこと**
- コーディングの際、リポジトリ全体構造を前提にした理解ができない
- 聞きたいことがあるとき、ファイルやコードスニペットを渡す必要がある
- 質問者の意図に寄せすぎて、設計の前提を深掘りしてくれないことがある

### Claude Code（Pro / Claude 3.5 $20）

**向いていること**
- リポジトリ全体を前提に理解してくれる
- 設定ファイルや構成を含めてまとめて作れる
- 技術的な是非をきちんと説明してくれる
- 必要ならNoと言ってくれる

**苦手なこと**
- トークン制限が厳しく、長く続けることはできない
- レスポンスが遅い

Claude Code一本化も検討しましたが、日常的な相談や思考整理にはレスポンスの早いChatGPTが欠かせず、この組み合わせに落ち着きました。

## 現在の開発フロー

現在は、以下のように役割を明確に分けて使っています。

| フェーズ | 内容 |
|---------|------|
| ① 調査 | ChatGPTに調べさせ、議論し、最終判断は自分で行う |
| ② 設計 | Claudeに設計書案を出させ、ChatGPTと並行相談しながら固める |
| ③ タスクリスト作成と実装案 | Claudeに分解と案を出させ、ChatGPTと中身を検討する |
| ④ 実装 | ファイル作成からClaudeに任せる |
| ⑤ コード理解・学習・レビュー | 不明点はChatGPTで噛み砕いて理解する |
| ⑥ 変更・修正 | 使い勝手を私がチェックし、変更・修正はClaude |

コマンドライン操作や初期セットアップ、ライブラリの追加は自分でやっています。AIに任せると、意図しないバージョンが入ったり、余計な設定が入ることがあるためです。リファクタする場合は、E2Eテストを先に書くといいでしょう。

## テンプレート

AIとの開発で使っているテンプレートを紹介します。

Plan.md、DesignIdea.mdなど複数ファイルに分けず、1ファイルにすべて入れています。1ファイルにまとめることで、ChatGPTに相談するときも渡しやすくなります。また、ファイルに残すことで、セッションが切れても続きから始められます。プロジェクトのリポジトリに入れるとGitHubでDiff確認時にノイズになるので、別の場所で保管しています。

:::details テンプレート構成（クリックで展開）
```
# ルール
（AIとの作業ルールを記載）

# 項目 ------------------
# 設計合意までのタスク
# 実装タスクリスト

# 設計書
## データ構造とデータフロー
## ワイヤーフレーム/画面構成
## プロジェクト構造
## 保存設計
## 状態管理
## モックデータ一覧
## ワークフロー
## アプリ設定 / 将来拡張ポイント

# 技術スタック
# 開発ルール
# レポ＆URL基本情報

# 人間エリア
# 人間ログ ------------------
# ログ ------------------
```
:::

技術スタックや開発ルールなど、プロジェクトをまたがって使うものは最初から指定しています。「人間エリア」と「ログ」には、私の必要なメモを入れています。

## これまでの経緯

使い始めた当初、Shift-JISからUTF-8への大量変換をAIに頼もうとしました。しかし、方法を探してもらうこともできず、AIが変換することもできず、結局、過去のPerl経験を頼りに自分でコードを書きました。

それでも、迷ったときに相談できる存在ができたことは大きかったです。友達のように会話しながら開発することが多くなりました。

一方で、問題が解決しないまま同じ提案を繰り返される「AIループ」にハマることも多くありました。コードを直せば済む問題でも、別のライブラリをインストールするよう言われるなど、納得できないこともありました。

## 設計が大切

とにかく作り始めて、問題が出たらその都度AIに相談して直せば何とかなると思っていました。しかし、場当たり的に修正を重ねると、AIは過去の文脈と新しい方針を混ぜてしまい、保守不能なコードになりました。

AIはコードを高速で量産できます。しかし、途中で設計が揺れると、プロジェクト全体は一気に崩れます。作りたいものを形にするには、最初に設計に落とすことが欠かせません。

## 設計のフェーズごとにテストすることの重要性

QAをやっていた経験があり、テストの重要性は理解していたつもりでした。でも、開発を始めると「テストはいつでもできる」と、後回しにしがちでした。AIを使うようになって、フェーズごとにきちんとテストして設計の範囲内で直すことの重要性を実感しています。

自分でテストすると、AIが提案してくれた一般的な方法ではなく、使い勝手などで自分が納得する形をとことん追求できます。AIは「正しそうな答え」は出せますが、「自分にとって正しい答え」は人間が決める必要があります。

テストの粒度を上げていくと、「ここをこうしたい」と思うことが出てきます。でも、そのために複雑な変更が必要なら、納得してばっさり切れるようになりました。シンプルさは大切にしています。

## 学び続けて、積極的に路線も変える

自分でコードを書いていたときは、「まあここは私の限界だな」などと諦めていました。今は、適切に設計し、対話し、責任を持つことで、AIが強力な開発パートナーになっています。

また、「設計」や「テスト」は重要ですが、結局のところ、自分の能力を超えたものは作れません。自分が理解できないことは、AIまかせになり、簡単に保守不可能になります。AIにコーディングを任せた分、技術や知識を学ぶ時間ができたと考えています。

今後もツールは変わっていくでしょうし、考え方やスタイルも変わっていくと思います。そのときは積極的に路線も変えていきたいと思います。
