AIを個人開発で使い始めて2年以上が経ちました。最初は「AIがあれば何でもできる」と思っていましたが、現実は違いました。試行錯誤を繰り返す中で気づいたのは、設計の重要性でした。

本記事では、2026年2月時点での私のAI活用スタイルを記録として残しておきます。

## これまでの経緯

2023年秋頃、個人開発でAIを使い始めました。当初は期待と裏腹に、Shift-JISからUTF-8への大量変換方法を探してもらうことができず、またAIが変換することもできず、結局、過去のPerl経験を頼りに自分でコードを書きました。

それでも、迷ったときに相談できる存在ができたことは大きかったです。友達のように会話しながら開発することが多くなりました。

一方で、問題が解決しないまま同じ提案を繰り返される「AIループ」にハマることも多く、また、うまく行かない場合にコードで直すことができるものでも、別のものをインストールするように言われるなど、納得できないこともありました。

2025年夏頃まで、この状態でした。その中で、ここ数ヶ月、今のスタイルに落ち着くことができました。

## 使っているAI

現在、2つのAIを併用しています。

### ChatGPT（Plus / GPT-5.2 $20）

**向いていること**
- 何でも気軽に相談できる
- レスポンスが早く、思考の壁打ちに向いている
- 技術以外の雑談・整理・文章化にも使いやすい

**苦手なこと**
- コーディングの際、リポジトリ全体構造を前提にした理解ができない
- ファイルを明示的に渡さないと文脈を失いやすい
- 質問者（私）の意図を汲むため、設計意図の説明が浅くなることがある

### Claude Code（Pro / Claude 3.5 $20）

**向いていること**
- リポジトリ全体を前提に理解してくれる
- 設定ファイルや構成を含めてまとめて作れる
- 技術的な是非をきちんと説明してくれる
- 必要ならNoと言ってくれる

**苦手なこと**
- トークン制限が厳しく、長く続けることはできない
- レスポンスが遅い

コーディング重視でプランをあげて、Claude Code一本化も検討しましたが、日常的な相談や思考整理にはレスポンスの早いChatGPTが欠かせないためこの２つを使っています。PerplexityやCopilotも検討しましたが、現時点ではこの組み合わせを使っています。

## 現在の開発フロー

| フェーズ | 内容 |
|---------|------|
| ① 調査 | ChatGPTに調べさせ、議論し、最終判断は自分で行う |
| ② 設計 | Claudeに設計書案を出させ、ChatGPTと並行相談しながら固める |
| ③ タスクリスト作成 | Claudeに分解させ、ChatGPTと優先度を調整する |
| ④ 実装案検討 | Claudeに案を出させ、ChatGPTと納得するまで対話する |
| ⑤ 実装 | Claudeに任せる。テスト案も出してもらう |
| ⑥ コード理解・学習 | 不明点はChatGPTで噛み砕いて理解する |
| ⑦ テスト・品質管理 | テストはすべて自分で。変更・修正はClaude |

コマンドライン操作や初期セットアップ、ライブラリの追加は自分でやっています。AIに任せると、意図しないバージョンが入ったり、余計な設定が入ることがあるためです。リファクタする場合は、上記手順でE2Eテストを書きます。

## テンプレート

AIとの開発で使っているテンプレートを紹介します。

Plan.md、DesignIdea.mdなど複数ファイルに分けて書いたこともありましたが、1ファイルにすべて入れる方が迷いなく進められます。また、プロジェクトのリポジトリに入れるとGitHubでDiff確認時にノイズになるので、別の場所で保管しています。

:::details テンプレート構成（クリックで展開）
```
# ルール
（AIとの作業ルールを記載）

# 項目 ------------------
# 設計合意までのタスク
# 実装タスクリスト

# 設計書
## データ構造とデータフロー
## ワイヤーフレーム/画面構成
## プロジェクト構造
## 保存設計
## 状態管理
## モックデータ一覧
## ワークフロー
## アプリ設定 / 将来拡張ポイント

# 技術スタック
# 開発ルール
# レポ＆URL基本情報

# 人間エリア
# 人間ログ ------------------
# ログ ------------------
```
:::

技術スタックや開発ルールなど、プロジェクトをまたがって使うものは最初から指定しています。「人間エリア」と「ログ」には、私の必要なメモを入れています。

## 設計が大切

AIはコードを高速で量産できます。しかし、途中で設計が揺れると、プロジェクト全体が一気に崩れます。

最初は、とにかく始めてみて、どんどん修正を入れて、詰まったらAIに手伝ってもらって解決できるものと思っていました。

しかし、場当たり的に修正を重ねると、AIは過去の文脈と新しい方針を混ぜてしまい、保守不能なコードになります。設計を固めずに実装に入ることは、むしろリスクが高いことに気がつきました。作りたいものを作ろうとすると、最初に設計に落とすことがいかに重要かがわかってきました。

コーディングの経験があると、設計を理解できます。でも、細かな点、また俯瞰する必要性など、まだまだ学ぶことはたくさんあります。今はこの学習に力をいれています。

## 自分でテストすることの重要性

かつてQAをやっていた経験から、テストの価値は理解していたつもりでした。でも、開発を始めると「テストはいつでもできる」と思い、後回しにしがちでした。AIを使うようになって、フェーズことにきちんとテストして設計の範囲内で直すことの重要性を実感しています。

E2Eテストなども書きますが、自分でテストすると、使い勝手など、AIが提案してくれた一般的な方法ではなく、自分が納得する変更をとことん話し合うことができます。

自分でコードを書いていたときは、「まあここは私の限界だな」などと諦めていましたが、今はやりたいことを相談しながらできるので、どんどん自分の理想が実現できます。

そして、ほんの小さな変更を入れるだけで、コードが複雑になると気がついた時は、ばっさり切ることで、シンプルさを追求することができるようになりました。（ただし、あとで何かに使えるかも、と思うと「人間エリア」に残したりしてます。）

## 変わらない原則

適切に設計し、対話し、責任を人間側が持つことで、AIは、強力な開発パートナーになります。

また、「設計」や「テスト」は重要ですが、結局のところ、今の自分の能力を超えたものは作れないということにも気がつきました。わからないことは、自分が理解できないとAIまかせになってしまい、簡単に保守不可能となります。だからこそ、技術や知識を入れていくことはこの後も追求していきたいと思っています。

今後もツールは変わるでしょう。でも、「設計重視」「最終判断は自分」という原則は変えずに使い続けていきます。
